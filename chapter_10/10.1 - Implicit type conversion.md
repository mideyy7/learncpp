## Key points

- **Why conversion is needed**: different types encode the "same" value with different bit patterns (e.g. `int` 3 vs `double` 3.0). The compiler can't just copy bits from one type's storage into another's — it must compute a converted value, then store *that*.
- **Implicit conversion** (a.k.a. automatic conversion / coercion) fires whenever an expression's type doesn't match the type required by context. It's the vast majority of conversions in real code. Contexts where it triggers:
  - Initializing/assigning a variable with a value of a different type: `double d{ 3 };`
  - Return value type differs from the function's declared return type.
  - Binary operators with mismatched operand types: `4.0 / 3` converts `3` to `double`.
  - Non-bool condition in an `if`: `if (5)` converts `5` to `bool`.
  - Function call argument type differs from parameter type.
- **The standard conversions**: a fixed set of core-language rules (14 as of C++23) the compiler consults to perform any conversion. Grouped into 5 categories:
  - **Numeric promotions** — small integral types → `int`/`unsigned int`, `float` → `double`. See 10.2.
  - **Numeric conversions** — other integral/floating conversions that aren't promotions. See 10.3.
  - **Qualification conversions** — add/remove `const`/`volatile`.
  - **Value transformations** — change value category (lvalue-to-rvalue, array/function-to-pointer decay, temporary materialization). See 12.2.
  - **Pointer conversions** — `nullptr` → pointer types, pointer → other pointer types.
  - This lesson doesn't require memorizing the full 14-rule table — the taxonomy matters more than the enumeration; numeric promotions/conversions (next two lessons) are the ones worth knowing cold.

## C++ internals

- Conversion is not reinterpretation: converting `int` 3 to `float` produces a *new bit pattern* representing 3.0, computed via the applicable standard conversion rule — it is not a bitwise copy/reinterpretation of the source object's storage.
- Contrast with `std::memcpy(&f, &n, sizeof(float))` copying an `int`'s raw bits into a `float` object: this is *not* a conversion, it's bit-level reinterpretation (legal, well-defined type punning via `memcpy`, or `std::bit_cast` in C++20+). The `float` then holds whatever value that bit pattern happens to encode under IEEE 754 — for `int` 3's bit pattern, that's roughly `4.2039e-45`, not `3.0`. This is the mental model check for "why can't the compiler just copy the bits."
- Every implicit/explicit conversion to a *value* (not a reference) produces a temporary object of the destination type holding the converted result; the original operand is untouched.

## Undefined behavior / gotchas

- `std::memcpy`/`std::bit_cast` type punning between same-sized types is **well-defined**, not UB — don't confuse "produces a nonsensical-looking value" with "undefined behavior." It's a deliberate bit-reinterpretation, standing in contrast to how implicit conversion actually works.
- Compilation fails when no standard conversion path exists between the source and target type (e.g. `int x { "14" };` — no `const char*` → `int` conversion).
- **Brace initialization narrows more strictly than the language otherwise allows**: `int x { 3.5 };` is a compile error even though the compiler *does* know how to convert `double` → `int` — brace-init specifically disallows conversions that can lose data. Direct/copy-init (`int x = 3.5;`) allows it silently.
- Overload resolution can be ambiguous when multiple standard conversions are equally viable candidates (covered in 11.3).

## Interview angle

- "Why doesn't the compiler just copy the bits when converting `int` to `float`?" — expect you to explain that different types have incompatible bit-level representations of the same logical value, so conversion must compute a new representation, not reinterpret an existing one. The `memcpy`-produced garbage float is the canonical illustration.
- Know the 5-category taxonomy (promotions, conversions, qualification, value transformation, pointer) well enough to place an example correctly — e.g. "is `int→long` a promotion or a conversion?" (conversion — only *integral promotions* target `int`/`unsigned int` specifically; widening beyond that is a numeric conversion).
- "Why does brace-init reject `int x{3.5}` but `int x = 3.5` compiles?" tests whether you know narrowing-conversion rejection is a brace-init-specific rule, not a general language rule — good example of "safety opt-in via syntax choice" in C++.
