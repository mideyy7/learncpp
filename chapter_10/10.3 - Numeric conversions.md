## Key points

- **Numeric conversion** ≠ **numeric promotion** ([[10.2 - Floating-point and integral promotion]]): promotions are a specific, always-safe subset (widening to `int`/`double` for efficient processing); anything else — including narrowing — is a numeric conversion. If a conversion is covered by the promotion rules, it's called a promotion, not a conversion.
- Five basic categories:
  1. Integral → any other integral type (excluding promotions): `int`→`short`, `int`→`unsigned int`, `short`→`char`.
  2. Floating point → any other floating point type (excluding promotions): `double`→`float`, `double`→`long double`.
  3. Floating point → integral: `double`→`int`.
  4. Integral → floating point: `int`→`double`.
  5. Integral or floating point → `bool`.

## Safety categories

- **Value-preserving**: destination type can represent every value of the source type exactly (`int`→`long`, `short`→`double`). Round-tripping back to the source type always reproduces the original value. No compiler warning.
- **Reinterpretive**: unsafe, but no data is lost — the bit pattern/value is reinterpreted, and converting back to the source type reproduces the original value even if the intermediate value was "wrong." Signed↔unsigned conversions are the main example (same mechanics as [[4.12 - Introduction to type conversion and static_cast]], applied implicitly here rather than via `static_cast`):
  - Signed→unsigned: in-range values convert unchanged; out-of-range (negative) values modulo-wrap to a large unsigned value.
  - As of C++20, signed integers are required to be 2's complement, so this wraparound is well-defined in both directions (signed→unsigned and unsigned→signed). Pre-C++20, out-of-range unsigned→signed was implementation-defined.
  - Compilers generally leave implicit signed/unsigned conversion warnings **off by default** — too many false positives (e.g. standard library container sizes are unsigned), so real bugs can slip through silently. Contrast with lossy conversions, which usually do warn.
- **Lossy**: data is actually discarded; round-tripping does *not* reproduce the original value.
  - `double`→`int`: fractional part truncated (`3.5`→`3`; converting `3` back to `double` gives `3.0`, not `3.5`).
  - `double`→`float`: precision loss (`1.23456789`→`1.23457`ish; the exact stored bits differ, not just the printed digits — see gotcha below).
  - Compilers generally *do* warn (sometimes error) on implicit lossy conversions.

## Undefined behavior / gotchas

- **Assigning outside destination range → garbage, not a clean error**: `int i{30000}; char c = i;` — `char` range is roughly -128..127, so this overflows the destination and produces an unexpected/implementation-defined-looking result (not necessarily related to `30000` in any obvious way).
- **Signed overflow is UB; unsigned "overflow" is well-defined wraparound** — same rule as [[4.5 - Unsigned integers, and why to avoid them]]. This applies to conversions too: converting into a signed type such that the result is out of that type's range was implementation-defined pre-C++20 / is now modulo-wrap-defined post-C++20 (per the reinterpretive rule above), but *arithmetic* overflow on signed types remains UB regardless of C++ version — don't conflate "conversion out-of-range" with "arithmetic overflow."
- **`int`→`double` is not guaranteed value-preserving** — it's safe on typical platforms (4-byte `int`, 8-byte `double`) but becomes lossy on architectures where both are 8 bytes (`double`'s 52-bit mantissa can't exactly hold all 64-bit integer values). Demonstrated by round-tripping a `long long` through `double`: `10000000000000001LL` → `double` → back to `long long` loses the last digit, becoming `10000000000000000`. Lesson: "value-preserving" is a property of the *platform's actual type sizes*, not the type names — don't assume portability.
- Floating point narrowing (`double`→`float`) loses actual precision bits, not just display precision — printed-digit differences you see with `std::setprecision` reflect a real difference in the stored value, not merely rounding at print time.

## Interview angle

- Be able to name the three safety buckets (value-preserving / reinterpretive / lossy) and give one example each — this is a clean way to show structured thinking about implicit conversions rather than "conversions are scary, be careful."
- "Why doesn't the compiler warn me about signed→unsigned conversions by default?" — good answer: it's reinterpretive, not lossy (no data loss, round-trips cleanly), and in practice most such conversions in real code (container `.size()`, indices) are harmless, so warning on all of them would drown real bugs in noise. Ties directly into the [[4.5 - Unsigned integers, and why to avoid them]] "prefer signed" argument.
- The `long long`→`double`→`long long` precision-loss example is a good concrete answer to "when is `int`-to-`double` not safe?" — shows awareness that safety of a conversion depends on relative type widths on the actual target platform, not a universal guarantee from the standard.
