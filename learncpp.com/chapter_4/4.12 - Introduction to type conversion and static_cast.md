## Key points

- **Type conversion** turns a value of one type into a value of another type. It does not modify the source variable/value — it produces a new (temporary) value, conceptually like a function call that takes the original value as input and returns a converted value.
- **Implicit conversion**: compiler converts automatically when types don't match (e.g. passing an `int` where a `double` parameter is expected). No syntax required at the call site.
- **Explicit conversion** (`static_cast<new_type>(expression)`): programmer explicitly requests the conversion and takes responsibility for the result — suppresses the compiler warning that would otherwise fire on a narrowing/lossy implicit conversion.
- Prefer `static_cast` over disabling "treat warnings as errors" — disabling warnings globally hides *all* future warnings, not just the expected one.
- Angle-bracket syntax (`<...>`) in C++ generally signals a parameterized type — `static_cast<T>` is the first example of this pattern (later: templates, `std::vector<T>`, etc.).
- Brace initialization enforces safety here: `int x { 5.5 };` is a compile error (narrowing conversion), forcing you to either pick a non-narrowing type or cast explicitly.

## Undefined behavior / gotchas

- **Narrowing conversions silently truncate**: `double` → `int` drops the fractional part (truncates toward zero, does not round). `static_cast<int>(5.5)` → `5`. This happens even for values with no fractional part (e.g. `5.0`) — the compiler still warns on the *implicit* form because it can't prove safety in general.
- **Sign conversions** (`static_cast` between signed/unsigned of same width):
  - If the value is representable in the destination type, it's unchanged — only the type changes.
  - If not representable and destination is **unsigned** → modulo-wrapped (well-defined). E.g. `static_cast<unsigned int>(-1)` → `4294967295` (assuming 32-bit).
  - If not representable and destination is **signed** → implementation-defined before C++20; modulo-wrapped as of C++20. E.g. `static_cast<int>(4294967295u)` → `-1` (C++20+), previously impl-defined.
- **`std::int8_t` / `std::uint8_t` (and their `_fast`/`_least` variants) are typically typedefs for `signed char` / `unsigned char`** on most implementations, not a distinct integer type:
  - `std::cout << myInt8` prints as a **character**, not a number (e.g. value `65` prints as `A`).
  - `std::cin >> myInt8` reads a **single character** from the stream, not a parsed integer — entering `"35"` extracts only `'3'` (ASCII 51) into the variable, leaving `'5'` in the stream buffer for a later extraction.
  - Fix: `static_cast<int>(myInt8)` for output; for input, read into a wider int type and range-check/cast down if you need `int8_t` semantics.
  - Not guaranteed behavior (implementation-defined whether `int8_t` is `char`-like), but true on most systems (GCC/Clang/MSVC on typical platforms).

## Interview angle

- A classic gotcha question: "why does `std::uint8_t x = 5; std::cout << x;` print an unusual result or behave like a char?" — tests whether you know fixed-width types are aliases, not guaranteed distinct types, at the smallest widths.
- Sign-conversion wraparound (`static_cast<unsigned int>(-1) == UINT_MAX`) is a common trick question root cause behind subtle bugs (e.g. a negative `size()` computation implicitly converted to `size_t`, producing a huge unsigned value that blows past a buffer or loop bound). Know it's *well-defined* modulo wrap for signed→unsigned, but was *impl-defined* for unsigned→signed pre-C++20 (now also modulo wrap).
- Good answer structure: distinguish "implicit conversion is a compiler-inserted equivalent of a function call producing a temporary" vs. "explicit `static_cast` is the same mechanism, just requested and warning-suppressed by the programmer" — shows you understand casts aren't reinterpreting bits (for value conversions) but computing a new value.
