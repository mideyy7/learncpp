## Key points

- An n-bit unsigned type has range `0` to `2^n - 1`, vs. a signed type's `-(2^(n-1))` to `2^(n-1) - 1` on the same bit width — both represent exactly `2^n` values, unsigned just shifts the whole range up so it starts at 0 instead of straddling it. This is *why* unsigned buys you "twice the positive range" for the same storage: it's not extra bits, it's the same bits repurposed.
- **Best practice (this site's stance, shared by e.g. Google, Bjarne Stroustrup): favor signed integers for quantities and arithmetic; avoid mixing signed/unsigned.** Controversial but widely advocated — the reasoning below (danger-zone placement + silent conversions) is *why*, not just an appeal to authority.
- Legitimate uses of unsigned, and the reason each is actually an exception rather than a counterexample to the rule:
  - Bit manipulation (masks/flags) — wraparound is irrelevant because you never treat the bits as a magnitude.
  - Algorithms wanting well-defined wraparound on purpose (encryption, PRNGs) — here wraparound *is* the feature, not a bug.
  - Array/container indexing (`.size()` returns `size_t`) — forced by the standard library's API, not a free choice.
  - Embedded/memory-constrained targets — the extra positive range or hardware bit-ops matter more than the safety tradeoff.

## C++ internals

- Wraparound is literal modulo arithmetic on the bit pattern, not a special case the compiler inserts: for an 8-bit unsigned type, `value mod 256` is stored directly because there's no sign bit reinterpretation happening — the bits are the value.
  ```cpp
  unsigned short x{ 65535 }; // 0xFFFF, largest 16-bit unsigned value
  x = 65536;                 // 65536 mod 65536 = 0
  x = 65537;                 // 65537 mod 65536 = 1
  x = -1;                    // -1 mod 65536 = 65535 (wraps from the top down)
  ```
- The **usual arithmetic conversions**: in a binary op with one signed and one unsigned operand of the same rank, the signed operand is implicitly converted to unsigned *before* the operation runs, and the result type is unsigned. This is ordinary implicit conversion (same mechanism as [[4.12 - Introduction to type conversion and static_cast]]) triggered automatically by the operator, not requested via `static_cast` — the danger is precisely that there's no cast token in the source marking where the conversion happens.

## Undefined behavior / gotchas

- **Unsigned wraparound is well-defined, not UB** — despite being colloquially called "overflow," the standard states unsigned computation can never overflow; it's `mod (max+1)` by definition. This matters because tooling (UBSan) can catch signed overflow but has nothing to flag here — a wraparound bug in unsigned code runs "correctly" as far as the language is concerned.
- **Unsigned subtraction underflow** — the single most common instance, because 0 (the danger zone) sits right where everyday non-negative values live, unlike signed overflow whose danger zones are far out at the extremes:
  ```cpp
  unsigned int x{ 2 };
  unsigned int y{ 3 };
  std::cout << x - y << '\n'; // 4294967295, not -1
  ```
- **Mixed signed/unsigned arithmetic silently converts the signed side**, so a subtraction that "looks" mixed-safe still wraps:
  ```cpp
  unsigned int u{ 2 };
  signed int   s{ 3 };
  std::cout << u - s << '\n'; // 4294967295 — s is converted to unsigned first
  ```
- **Mixed signed/unsigned comparison can silently invert the obvious answer**, because the signed operand is converted before the comparison, not after:
  ```cpp
  signed int   s{ -1 };
  unsigned int u{ 1 };
  if (s < u)                       // -1 becomes 4294967295, so 4294967295 < 1 is false
      std::cout << "-1 is less than 1\n";
  else
      std::cout << "1 is less than -1\n"; // this branch runs
  ```
- **Passing a negative value to an unsigned parameter wraps at the call boundary, usually silently**:
  ```cpp
  void doSomething(unsigned int x) { std::cout << "x is " << x << '\n'; }
  doSomething(-1); // prints 4294967295; no warning without aggressive sign-conversion flags
  ```
- Compilers do warn on signed/unsigned mismatches in many of the cases above, but they warn identically on harmless mixed-sign expressions (e.g. both operands provably positive), so the noise buries the real bugs unless you're specifically scanning for this class of warning.

## Interview angle

- **Q: "Why does this loop never terminate?"**
  ```cpp
  for (size_t i = size - 1; i >= 0; --i) { /* ... */ }
  ```
  **A:** `size_t` is unsigned, so `i >= 0` is a tautology — there's no representable negative value for `i` to reach. When the loop body decrements `i` from `0`, it doesn't go to `-1`, it wraps to `SIZE_MAX` (well-defined modulo wraparound, per the mechanism above), and the loop keeps running, now reading/writing far out of bounds instead of stopping. Fix: iterate signed, or restructure as `for (size_t i = size; i-- > 0; )` so the decrement happens before the unsigned value is compared as a stopping condition it can actually satisfy at 0.

- **Q: "This code compiles, looks obviously correct, and prints the wrong branch. Why?"**
  ```cpp
  signed int s{ -1 };
  unsigned int u{ 1 };
  if (s < u) std::cout << "-1 is less than 1\n";
  else       std::cout << "1 is less than -1\n";
  ```
  **A:** The usual arithmetic conversions kick in on `<`: since one operand is unsigned, `s` is implicitly converted to `unsigned int` *before* the comparison, turning `-1` into `4294967295`. The comparison actually evaluated is `4294967295 < 1`, which is false, so the `else` branch runs. The fix isn't "be careful" — it's to never let a signed and unsigned value meet in the same expression; cast explicitly at the boundary (`static_cast<int>(u)` or vice versa, after confirming range) so the conversion is visible in the source instead of implicit.

- **Q: "When would you actually choose unsigned, given all this?"**
  **A:** Only where wraparound is the point (bit masks, PRNGs/hashing) or where you're forced into it by an API (`.size()`, indices). Never for "this quantity happens to always be non-negative" — that's exactly the trap, because the moment an intermediate computation (a subtraction, a decrement past zero) produces a conceptually-negative value, unsigned turns it into a huge positive one instead of a value the compiler/sanitizer can flag as wrong.
