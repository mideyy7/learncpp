## Key points

- The **usual arithmetic conversions** are the formal rule set behind something already used informally in [[4.5 - Unsigned integers, and why to avoid them]]: whenever a binary operator that requires matching operand types (`+ - * / %`, relational `< > <= >= == !=`, bitwise `& ^ |`, and `?:`'s two result operands) gets mismatched operand types, one or both operands are implicitly converted to a shared **common type** before the op runs.
- **Overloaded operators are exempt** — the usual arithmetic conversions only apply to the builtin operators on fundamental types; a user-defined `operator+` follows whatever overload resolution picks, not this ranking.
- Rank order (high to low): `long double > double > float > long long > long > int`. Anything below `int` in width (e.g. `short`, `char`) isn't even on the list — it's promoted to `int` first via integral promotion (10.2), independent of this ranking.
- The algorithm is two sequential steps, and the order matters:
  1. **Int-vs-float beats promotion**: if one operand is integral and the other floating-point, the integral one converts directly to the floating-point type — no integral promotion happens first. Otherwise, any operand below `int` rank gets promoted to `int` (or `unsigned int`) per 10.2 before rank-matching even starts.
  2. **Then rank-match**: if signs still differ post-promotion, the special signed/unsigned rules below apply; otherwise the lower-rank operand converts up to the higher-rank operand's type.
- `std::common_type_t<A, B>` (`<type_traits>`) computes the same result this algorithm would produce, as a compile-time query — useful whenever you need to declare a variable or return type that matches what a mixed-type expression would deduce (e.g. a function returning either of two numeric parameter types).

## C++ internals

- The signed/unsigned resolution (step 2, when ranks or signs conflict) is a 3-way fallback, not "whichever is unsigned wins":
  1. If the unsigned operand's rank ≥ the signed operand's rank → signed converts to that unsigned type.
  2. Else, if the signed type can represent every value the unsigned type can (i.e. signed has strictly higher rank) → unsigned converts to the signed type.
  3. Else (ranks are close enough that neither type fully covers the other) → **both** convert to the unsigned counterpart of the signed operand's type.
  - Case 1 is what fires in the classic `int`/`unsigned int` gotchas below, since same-rank unsigned beats signed.
- `typeid(expr).name()` (`<typeinfo>`) reflects exactly which common type the conversion algorithm picked — it's the debugging tool for "what type did this expression actually become," which is otherwise invisible at the call site since the conversion is fully implicit.

## Undefined behavior / gotchas

- **`unsigned - larger-magnitude-signed` wraps silently**, because the `int` literal is promoted to `unsigned int` (same rank, unsigned wins) before the subtraction runs — this is the mechanism instance of the wraparound already covered in [[4.5 - Unsigned integers, and why to avoid them]], but here it's triggered purely by literal suffixes/types meeting in an expression, with no explicit cast anywhere:
  ```cpp
  std::cout << typeid(5u - 10).name() << ' ' << 5u - 10 << '\n';
  // unsigned int 4294967291   (NOT -5 — 10 became unsigned before subtracting)
  ```
- **Signed/unsigned relational comparisons can silently invert**, same root cause — the signed side converts to unsigned before `<` evaluates, so a negative number can compare "greater than" a small positive one:
  ```cpp
  std::cout << std::boolalpha << (-3 < 5u) << '\n';
  // false — (-3) is converted to a huge unsigned value first, which is NOT < 5u
  ```
- **No compiler warning is guaranteed for either case above** — the conversion is a fully standard-sanctioned implicit conversion (not a narrowing one caught by brace-init rules), so it's easy to ship silently. This is the concrete mechanism that makes "avoid mixing signed/unsigned" a rule about *expressions*, not just variable declarations — a bare integer literal like `10` or `-3` is enough to trigger it against any `unsigned` operand.

## Interview angle

- **Q: "What does `typeid(5u - 10).name()` print, and why isn't it `-5`?"**
  **A:** `5u` is `unsigned int`; `10` is `int`, same rank. Per the usual arithmetic conversions' signed/unsigned step, when ranks are equal the unsigned operand wins, so `10` converts to `unsigned int` *before* the subtraction executes. The computed expression is really `5u - 10u`, which wraps (well-defined modulo behavior, per [[4.5 - Unsigned integers, and why to avoid them]]) to `4294967291`, not a negative number that then gets reinterpreted — the value was never negative at any point in the computation.

- **Q: "Walk through why `-3 < 5u` evaluates to `false`."**
  **A:** The relational operator `<` is one of the operators requiring matching operand types, so the usual arithmetic conversions run before the comparison: `-3` (signed) and `5u` (unsigned) are same rank, so `-3` converts to `unsigned int` first, producing a value near `UINT_MAX` (specifically `2^32 - 3`). The comparison actually performed is `(2^32 - 3) < 5`, which is false. The bug isn't in the `<` operator's logic — it's that the operands were converted to a shared type *before* the comparison had a chance to compare "signed vs unsigned" semantics at all.

- **Q: "How would you write a function that returns whichever type a mixed `int`/`double` expression would produce, without hardcoding `double`?"**
  **A:** Use `std::common_type_t<T1, T2>` as the return type (or for a local variable) — it queries the same promotion/conversion ranking the compiler applies to binary operators, so `std::common_type_t<int, double>` is `double`, and `std::common_type_t<unsigned int, long>` follows the same rank/signedness resolution described above. This keeps the declared type in sync with the language's own conversion rules instead of the author guessing and hardcoding a type that might not match on another platform (e.g. where `long` and `int` have different widths).
